# Конфетная фабрика

Онлайн-табло для командной игры-соревнования по решению задач. Команды
покупают задачи за внутриигровую валюту, решают их и получают бонусы; на
экране в реальном времени растут «запасы» и меняется таблица задач. Решения
могут подтягиваться автоматически с [informatics.msk.ru](https://informatics.msk.ru)
или засчитываться вручную/по ответу — в зависимости от режима игры.

Приложение — один самодостаточный бинарник на Go: HTTP-сервер со страницами и
JSON-API, фоновый опросчик информатикса и файловая база SQLite. Внешних
зависимостей во время работы нет — HTML, CSS, JS и звуки встроены в бинарник.

---

## Как это работает: правила игры

У каждой команды одинаковый стартовый капитал: **запасы** конфет и **скорость**
их производства. Каждую игровую секунду запасы растут на текущую скорость.
Задачи разложены по сетке `n×n` (n уровней сложности, по n задач в уровне),
порядок задач внутри уровня у каждой команды перемешан индивидуально.

- **Покупка задачи** — списывает «цену задачи» из запасов и временно снижает
  скорость на «нагрузку». Условие задачи открывается только после покупки.
- **Покупка теста (подсказки)** — списывает «цену подсказки». Тестов можно
  купить сколько угодно.
- **Решение задачи** — возвращает нагрузку, добавляет «бонус к скорости»
  (навсегда) и «бонус к запасам». Момент зачёта — время отправки решения.
- Побеждает команда с наибольшими запасами в момент окончания игры.

Цены и бонусы задаются по уровням при создании игры. Значения по умолчанию и
советы по балансировке — в [docs/balance-guide.pdf](docs/balance-guide.pdf).

### Ключевая идея: журнал событий

Все действия (покупки, тесты, решения) хранятся как **журнал событий**
(тип + время). Текущее состояние нигде не хранится — оно каждый раз
детерминированно пересчитывается из журнала. Поэтому ведущий может исправить
любую ошибку, отредактировав журнал (сдвинуть время, удалить, отключить или
добавить событие задним числом), и цифры на всех экранах мгновенно
пересчитаются правильно.

## Режимы игры

Режим выбирается при создании и не меняется после старта.

- **Informatics** — решения подтягиваются автоматически по аккаунтам команд на
  информатиксе. Организатор задаёт ссылки на задачи и `user_id` команд. Покупки
  вводит организатор либо команды сами со своей страницы.
- **Ручной / математический** — информатикс не используется. Задаётся число
  уровней и, для каждой из `n×n` задач, необязательное **условие** (ссылка или
  текст, открывается команде после покупки) и **эталонный ответ**. Команды сами
  покупают задачи и **отправляют ответы**: совпал с эталоном (с точностью до
  пробелов и регистра) — задача засчитана, нет — «Неверно» и короткий сигнал.
  Задачи без эталона засчитывает организатор вручную.

## Роли и страницы

| Кто | Где | Что видит и делает |
|-----|-----|--------------------|
| **Зритель** | `/g/{id}` | Публичное табло: запасы, скорость, сетка задач по цветам (не куплена / куплена / решена), гистограмма, таймер. Без ссылок на задачи. |
| **Команда** | `/g/{id}/team` (логин/пароль) | То же табло плюс своя таблица задач: покупка, условие после покупки, отправка ответа (в мат-режиме), время сдачи решённых задач. |
| **Организатор** | `/admin` (логин/пароль) | Создание и настройка игр, живое табло с ручным вводом действий, журнал событий, аномалии, статус опросчика, оформление сайта. |

Цвета ячеек: серая — не куплена, жёлтая — куплена, зелёная — решена (с галочкой
и временем сдачи от старта в формате Ч:ММ). Страницы обновляются сами; при
решении задачи звучит сигнал и разлетаются конфеты, при финише — объявление
победителя и салют.

### Оформление

Три темы, переключаются в админке и применяются ко всему сайту сразу:
🍬 «Конфетная фабрика», 🤖 «Токены и нейросети», 🐹 «Тапаем хомяка». Для каждой
есть свой свод правил для игроков: [docs/rules-*.pdf](docs/).

## Автозачёт с информатикса

Фоновый опросчик раз в `poll-interval` обходит аккаунты команд активных игр и
подтягивает новые решённые посылки (вердикт OK или «Зачтено»).

- Посылка сопоставляется с задачей игры по `chapterid` ссылки и с командой по
  `user_id`. Совпала, задача куплена, время в пределах игры — создаётся событие
  «решение» со временем посылки.
- Иначе посылка попадает в **аномалии** (задача не куплена / уже решена / вне
  времени игры). Ведущий разбирает их кнопками «Принять» / «Отклонить».
- Опрос продолжается ещё 15 минут после конца игры — чтобы зачесть посылки,
  отправленные «на флажке» и проверенные с задержкой.
- Посылки, которые ещё тестируются, не пропускаются: их поздний вердикт будет
  дождан (при этом следующий цикл опроса делается раньше).
- Кнопка **«Проверить решения»** (и автопроверка при сохранении конфигурации)
  позволяет ещё до старта увидеть, не решены ли задачи командами заранее.

Секреты сервисного аккаунта никогда не попадают в логи или HTML. Если аккаунт
не настроен, веб-часть работает, а в админке висит баннер с ошибкой опросчика.

## Запуск локально

Нужен Go 1.25+ (CGo не требуется — драйвер SQLite на чистом Go). Более старый
тулчейн Go подтянет нужную версию сам.

```sh
go build -o candy-factory .
./candy-factory                 # слушает :8080, данные в ./data
```

При первом запуске создаётся каталог `./data` и файл
`data/credentials/admin_credentials.json` с логином `admin` и паролем
`change_me` — **обязательно поменяйте**. Файл читается при каждом входе, рестарт
не нужен.

Откройте `http://localhost:8080/admin`, войдите, создайте игру.

Для автозачёта с информатикса заполните
`data/credentials/informatics_credentials.json` (пример — в
[data.example/](data.example/)):

```json
{
  "username": "сервисный_логин",
  "password": "пароль",
  "base_url": "https://informatics.msk.ru"
}
```

`base_url` можно указать зеркалом `https://informatics.mccme.ru`.

### Параметры запуска

Задаются флагами (или переменными окружения `CANDY_LISTEN`, `CANDY_DATA_DIR`):

| Флаг | По умолчанию | Значение |
|------|--------------|----------|
| `-listen` | `:8080` | адрес HTTP-сервера |
| `-data-dir` | `./data` | каталог данных |
| `-poll-interval` | `30s` | период опроса информатикса |
| `-page-refresh` | `1s` | период автообновления страниц |

Время на страницах показывается в часовом поясе сервера — задаётся переменной
`TZ` (например, `TZ=Europe/Moscow`).

### Каталог данных

```
data/
  app.db                          # база SQLite (создаётся и мигрируется сама)
  secret                          # ключ подписи cookie (генерируется сам)
  theme.txt                       # выбранное оформление сайта
  credentials/
    admin_credentials.json        # {"login": "...", "password": "..."}
    informatics_credentials.json  # сервисный аккаунт информатикса
  cache/
    informatics_state.json        # кеш опросчика (позиции выгрузки)
  server.log                      # текстовый лог
```

При рестарте (в т. ч. после падения) состояние полностью восстанавливается из
базы: идущие игры продолжаются, решения не задваиваются. Схема БД мигрирует
автоматически при старте — обновление версии не требует ручных действий.

## Развёртывание на сервере (Ubuntu)

```sh
# сборка (можно на своей машине под сервер)
GOOS=linux GOARCH=amd64 CGO_ENABLED=0 go build -o candy-factory .

# установка
sudo useradd -r -s /usr/sbin/nologin candy
sudo mkdir -p /opt/candy-factory
sudo cp candy-factory /opt/candy-factory/
sudo cp -r data.example /opt/candy-factory/data
sudo chown -R candy:candy /opt/candy-factory
# отредактируйте пароли в /opt/candy-factory/data/credentials/*.json

# systemd (unit-файл — в deploy/, Restart=always)
sudo cp deploy/candy-factory.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now candy-factory
```

Приложение слушает обычный HTTP; HTTPS даёт reverse-proxy перед ним. Примеры
конфигов nginx — в [deploy/](deploy/): `nginx.conf.example` (с доменом и
Let’s Encrypt) и `nginx-ip-selfsigned.conf` (доступ по IP с самоподписанным
сертификатом).

### Обновление версии

Собрать новый бинарник, заменить и перезапустить — данные и каталог не трогаются:

```sh
sudo systemctl stop candy-factory
sudo cp candy-factory /opt/candy-factory/candy-factory
sudo systemctl start candy-factory
```

Открытым страницам в браузере может понадобиться жёсткое обновление
(Ctrl+Shift+R), чтобы подтянулись новые стили и скрипты.

## Проведение игры: коротко

1. `/admin` → «Создать игру»: название, режим, число уровней, параметры уровней
   (предзаполнены), стартовые запасы/скорость, длительность, задачи и команды.
   Ошибки в ссылках и дубли задач подсвечиваются сразу.
2. Раздайте командам ссылки на их страницы и пароли (секция «Команды»).
3. «Старт сейчас» или плановое время старта (игра стартует автоматически).
   После старта состав задач и число уровней не меняются, остальное — можно.
4. Во время игры: покупки и решения вводятся кликом по ячейке живого табло либо
   командами сами; «Продлить» меняет длительность; ошибки правятся через журнал
   событий. Решения с информатикса и аномалии появляются автоматически.
5. По окончании — «В архив».

## Разработка

```sh
go test ./...     # юнит- и интеграционные тесты
go vet ./...
```

Структура: `internal/store` — SQLite и модели; `internal/game` — расчёт
состояния из журнала; `internal/informatics` — клиент и опросчик информатикса;
`internal/answer` — сверка ответов мат-режима; `internal/links` — нормализация
ссылок; `internal/web` — HTTP-сервер, страницы и API (шаблоны и статика
встроены через `embed`).
